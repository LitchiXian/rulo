# 接下来该做什么

## 第一优先级：把权限和菜单真正做闭环

### 1. 动态路由 + 动态菜单
当前 `rulo-admin` 的菜单和路由还是静态写死的。

要做的事：
- 登录后根据后端返回的菜单数据动态生成路由
- 侧边栏从菜单数据渲染，而不是模板写死
- 区分目录 / 菜单 / 按钮权限

为什么最优先：
- 这是管理系统和“演示后台”的分水岭
- 后面 RBAC、菜单管理、权限控制都依赖这个闭环

## 第二优先级：把 RBAC 做实，不只是 CRUD

### 2. 角色-菜单 / 角色-权限关联
当前已经有用户、角色、菜单、权限四个基础模块，但“谁能看到什么、谁能点什么”这件事还没有真正打通。

要做的事：
- 增加角色绑定菜单接口
- 增加角色绑定权限接口
- 登录后按当前用户角色返回菜单树和权限列表
- 前端按钮级权限控制（如 `v-permission` 或统一判断函数）

为什么现在做：
- 这是你后台最核心的业务能力
- 做完它，整套系统才真正像“管理系统”

## 第三优先级：补分页，不然后面列表都会越来越重

### 3. 通用分页查询封装
现在列表接口大多还是直接全量查。

要做的事：
- 后端补 `PageRequest / PageResponse`
- 用户 / 角色 / 菜单 / 权限列表支持分页和条件查询
- 前端表格接分页器

为什么现在做：
- 这是最容易在后期引发重构的点
- 越晚做，改动面越大

## 第四优先级：把个人中心从“前端占位”补成正式能力

### 4. 个人中心后端字段支持
当前个人中心里：
- 基础信息和密码已经可以走现有 update 接口
- 头像和背景图还是前端本地存储

要做的事：
- 后端给 `sys_user` 增加头像字段
- 后续如需要，再加个人主页背景字段
- 前端改为保存到后端而不是 localStorage

为什么不是最优先：
- 不影响系统主流程
- 但它是用户体验的完整性补丁，适合在 RBAC / 分页之后跟进

## 第五优先级：补部署与健康检查

### 5. Docker + `/health`
你现在功能越来越多，后面本地和部署环境差异会越来越明显。

要做的事：
- 增加 `GET /health`
- 检查服务 / PostgreSQL / Redis 状态
- 写 `Dockerfile`
- 写 `docker-compose.yml`

为什么现在值得做：
- 这会直接提升项目的“可运行性”和“可演示性”
- 以后你自己重装环境、换机器、部署测试都会轻松很多
